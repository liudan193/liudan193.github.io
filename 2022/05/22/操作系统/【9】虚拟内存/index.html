【9】虚拟内存

发表于2022-05-22|更新于2022-05-23|操作系统
|阅读量:

背景

问题

代码需要在内存中执行，但有些代码部分整个程序很少使用

  • 错误代码、异常例程、大型数据结构

不需要同时使用整个程序代码

仅加载执行部分程序

  • 程序不再受物理内存的限制
  • 每个程序运行时占用更少的内存 -> 更多的程序在同一时间运行
  • 提高了CPU利用率和吞吐量，但没有增加响应时间或周转时间
  • 将程序加载或交换到内存中所需的I/O更少->每个用户程序运行更快

虚拟内存

虚拟内存 —用户逻辑内存与物理内存的分离

  • 只有部分程序需要在内存中执行
  • 因此逻辑地址空间可以比物理地址空间大得多
  • 允许多个进程共享地址空间
  • 允许更高效的进程创建
  • 更多程序同时运行
  • 加载或交换进程所需的I/O更少

虚拟地址空间

虚拟地址空间 —进程如何存储在内存中的逻辑视图

  • 通常从地址0开始，连续地址直到空间结束
  • 同时，物理内存组织在页框中
  • MMU 必须将逻辑地址映射到物理地址

虚拟地址空间通常为堆栈设计逻辑地址空间，使其从最大逻辑地址开始，并在堆“向上”时“向下”增长

  • 最大限度地利用地址空间
  • 两者之间未使用的地址空间为空
  • 在堆或堆栈增长到给定的新页之前，不需要物理内存

启用稀疏地址空间，并留有可供增长的孔、动态链接的库等
通过映射到虚拟地址空间共享系统库
通过将页面读写映射到虚拟地址空间来共享内存
可以在fork()期间共享页面，从而加快进程创建

实现方法

虚拟内存可以通过以下方式实现：

  • 请求分页调度（Demand paging）
  • 请求分段调度（Demand segmentation）

请求分页

从磁盘加载可执行程序到内存中有哪些方法：

  1. 一是在程序执行的时候，将整个程序加载到物理内存
  2. 二就是请求分页，只有需要的时候才加载页面，这种技术常常用在虚拟内存管理技术中

请求分页系统在进程需要执行的时候，将进程加载到内存中，但是不是一整个进程，而是采用惰性交换器，仅仅加载所需的页面（处理页面的转换器是调页程序）。

基础概念

硬件支持

页表：能够通过有效-无效位或者保护位的特定值将条目标记为无效；
外存：或者说是辅助存储，这种外存用于保存不在内存中的那些页面；外存通常为高速硬盘，称为交换设备；用于交换的这部分称为交换空间；

有效-无效位

怎么知道该页在不在内存呢？
可以通过设置有效-无效位，来实现判断：

v：在内存中，叫做内存驻留
i：不在内存中，或者不在逻辑地址空间内

内存驻留： 如果所需的页面已经驻留在内存中，那么与非请求分页没有区别

缺页错误： 如果需要页面且不在内存中，需要检测页面并将其从存储加载到内存中，不改变程序行为，程序员无需更改代码

缺页错误（缺页中断）

在MMU地址转换期间，如果页表项中的有效-无效位为i，就会产生缺页错误

处理缺页错误的程序：

  1. 检查这个进程的内部表，确定该引用是有效的还是无效的；
  2. 如果引用无效，那么终止进程；如果引用有效但是尚未调入页面，那么应该去调入；
  3. 找到一个空闲帧；
  4. 调度磁盘，将所需的页面分配到刚分配的帧；
  5. 重置页表以指示内存中的页面，设置 validation bit = v；
  6. 重新执行导致页面中断的指令

纯按需分页

极端的一种情况 – 启动进程时内存中没有页面

操作系统将指令指针设置为进程的第一条指令, 非内存驻留 -> 缺页错误

程序的局部访问性

实际上, 给定的指令可以访问多个页面 -> 产生多个缺页错误

这样会导致性能下降，因为局部访问性(locality of reference)开销减少

请求分页的性能

设p为发生缺页错误的概率

有效访问时间 = (1-p)*内存访问时间 + p*缺页错误时间

为了计算缺页错误时间，需要知道他的所有动作：

  1. 操作系统中断
  2. 保存用户注册信息和进程状态
  3. 确定中断是页面中断
  4. 检查页面访问是否合法，并确定页面在磁盘上的位置
  5. 从磁盘读取一个空闲帧：
    5.1. 在队列中等待该设备，直到读取请求得到服务
    5.2. 等待设备寻道或延迟时间
    5.3. 开始将页面装载到空闲的页帧内
  6. 在等待时，将CPU分配给其他用户
  7. 从磁盘I/O子系统接收中断（I/O完成）
  8. 为其他用户保存寄存器和进程状态
  9. 确定中断来自磁盘
  10. 更正页表和其他表以显示页面现在在内存中
  11. 等待CPU再次分配给这个进程
  12. 恢复用户寄存器、进程状态和新页表，然后恢复被中断的指令

在任何情况下，缺页错误的处理时间主要有是三个主要组成部分：

  1. 处理缺页错误中断；
  2. 读入页面；
  3. 重新启动进程；

写时复制

写时复制是请求分页的优化，写时复制就是在写的时候才复制

写时复制 (COW) 允许父进程和子进程最初在内存中共享相同的页面

  • 如果任一进程写入共享页面，那么该页面才会被复制
  • 只是复制写入的那一页，而不是全部进程
  • 写入时复制是多种操作系统（包括 Windows XP、Linux 和 Solaris）使用的常用技术

空闲页面的分配

需要注意的一点是空闲页面的分配位置：通常，空闲页面是从按需零填充页面池中分配的，这个池应始终具有空闲帧以快速执行请求页面

vfork()操作

使用fork()的变种vfork()，使父进程挂起，子进程写时复制父进程的地址空间，这样的话，父进程恢复的时候，子进程写入的东西父进程是可见的；好处是不用复制页面，非常高效；

页面置换

注意：
请求分页是发现进程不在内存中，将外存的页放到内存；
而页面置换是指，当内存已经满了，应该将哪些页换出内存；

通过修改页面错误服务例程以包括页面替换来防止内存过度分配

I/O 缓冲区也消耗大量内存

页面替换完成了逻辑内存和物理内存之间的分离——可以在较小的物理内存上提供大的虚拟内存

基本页面置换

  1. 在磁盘上找到所需页面的位置
  2. 找到一个空闲帧：
    a.如果有空闲帧，那么就使用它
    b. 如果没有空闲帧，那么就是用页置换算法选择一个“牺牲”帧
    c. 将牺牲帧的内容写入磁盘上
  3. 将所需页面放入（新）空闲页帧中，更新页表和帧表
  4. 重新执行导致中断的指令，继续该进程

如果没有空闲帧，则需要进行两次页面传输（一出一入）
可以使用修改位减少时间：
修改位（脏位）：由于内存中的每一页都在外存中存放有副本，因此，若未被修改，则置换出该页的时候，就不需要将该页放回外存，节省一次时间

不会起名

为了完成请求调页，必须实现帧分配算法和页面置换算法

帧分配算法，需要确定：

  • 每个进程给多少帧
  • 置换哪些帧

页面置换算法：

  • 想要首次访问和重新访问时缺页率最低

下面讲页面置换算法：

页面置换算法

页面置换算法主要是要缺页率最低；

缺页率与引用串和可用帧的数量有关；

引用串就是程序一段时间内需要调用的页号序列
比如引用串：7,0,1,2,0,3,0,4,2,3,0,3,0,3,2,1,2,0,1,7,0,1
指的是程序先使用第7页的指令，再执行第0页的指令，再第1页……

FIFO页面置换

介绍：选择最旧的页面置换出去；

实现：只需使用FIFO队列实现；

Belady异常： 帧数增加，缺页错误率反而更大

最优页面置换

介绍：替换最长时间不使用的页面；

实现：需要预知以后要读的页面；

这个算法是最优的，他的缺页错误率在所有算法中最低，可以把他作为上限评价其他置换算法；

他不会遭受Belady异常；

LRU页面置换（最近最少使用算法）

介绍：替换最长时间没有被使用过的页面；

实现：有两种方法：

  1. 计数器
    每个页面条目都有一个计数器；每次通过此条目引用页面时，将时钟复制到计数器中
    当需要更改页面时，查看计数器以找到最小值
  2. 堆栈
    以双向链表形式保留页码堆栈：
    访问页面：将其移至顶部、需要更改 6 个指针
    但每次更新都会更加昂贵
    不搜索替换

需要额外的硬件；

没有Belady异常；

近似LRU页面置换

没有硬件支持的时候，用不了LRU，可以使用近似的LRU；

通过引用位实现，每当引用一个页面时，他的页面引用位就被硬件置位，页表中的每个条目联系这一个引用位；

最初，与每个页面关联一位，初始值=0，当页面被访问时，位设置为1，用访问位 = 0 的替换（虽然不知道顺序）；

二次机会算法

一般为 FIFO，外加硬件提供的访问位；

如果要替换的页面：
访问位 = 0，则替换它；
访问位 = 1，则将访问位设为 0，将页面留在内存中，置换下一个页面，遵循相同的规则；

增强型的二次机会算法

通过使用访问位和修改位（如果可用）协同改进算法

取有序对（参考、修改）

  1. (0，0)最近未使用未修改——最佳置换页面
  2. (0，1)最近未使用但已修改——不太好，必须在置换前写出
  3. (1，0)最近使用但未修改——可能很快会再次使用
  4. (1，1)最近使用且已修改——可能很快会再次使用，需要在更换之前写出

当需要页面置换时，使用时钟方案置换在最低非空类型中所碰到的页，可能需要多次搜索循环队列

帧分配

帧的最小数

每个进程需要的帧的最少数量
最小数量由计算机体系结构定义

计算机体系结构所需的最小数量（ISA）示例：IBM 370 – 6 个页面来处理 SS MOVE 指令：
指令是6字节，可能跨越2页
2页用于处理要移动的字符块
2页用于移动到目的区域的字符块

最大值当然是系统中的总帧数
两个主要的分配方案：

  • 固定分配
  • 优先级分配

分配算法

固定分配

平均分配 – 例如，如果有100个页帧(为操作系统分配页帧后)和5个进程，给每个进程20个帧；不能整除可以将剩下的作为空闲帧缓冲池；

按比例分配 – 根据进程的大小进行分配；
随着多道程序程度的变化，进程大小也在变化；

优先级分配

优先级分配 - 使用优先级分配方案，此方案根据优先级而不是比例；
如果进程Pi 产生一个缺页中断，选择置换其中一个帧，从优先级较低的进程中选择一个帧进行置换

全局分配 vs. 局部分配

全局置换 – 进程从所有帧集合中选择一个进行置换，一个进程可以从其他进程获取帧

  1. 但是随后流程执行时间可能会有很大差异
  2. 通常会导致更大的系统吞吐量，因此是更常用的方法

局部置换 – 每个进程仅从它自己的一组分配帧中进行选择

  1. 每个进程有更一致的性能
  2. 但可能未充分利用内存

非均匀内存访问

到目前为止，所有的内存访问是平均的

具有明显不同的内存访问时间的系统称为非均匀内存访问（NUMA）系统；

许多系统都是NUMA系统 – 多个CPU访问内存的速度各不相同

  1. 考虑包含CPU和内存的系统板，通过系统总线互连
  2. 最大限度地减少整体内存延迟并最大限度地提高 CPU 缓存命中率

最佳性能来自分配内存“接近”调度线程的 CPU

  1. 并尽可能修改调度程序以在同一系统板上调度线程
  2. 由Solaris通过创建用户组来解决
    • 跟踪 CPU/内存低延迟组的结构
    • 使用自己的调度器和调页程序
    • 尽可能调度一个进程的所有线程，并在组内为该进程分配所有内存

系统抖动

如果一个进程没有“足够”的页面，缺页率就会很高，就会不断的置换帧；

如果一个系统的调页时间多于他的执行时间，那么这个进程就在抖动；

因此，要防止抖动，否则对CPU性能有很大的影响；

工作集模型

为了防止抖动，应该为进程提供足够多的所需帧数，但是如何知道进程需要多少帧呢？工作群策略研究一个进程实际上使用多少帧；

局部性模型

工作集模型这种方法定义了局部性模型：
局部性模型指出，随着进程执行，进程从一个局部移向另一个；
一个程序通常由多个不同的可能重叠的局部组成；

局部是由程序结构和数据结构来定义的；

如果没有能够分配足够的帧给当前的当前的局部，就会发生抖动，因为不能在内存中保留正在使用的所有页面；

工作集模型

工作集模型是基于局部性假设的；

工作集窗口△：检查最近△个页面引用，这△个页面引用的集合称为一个工作集；工作集是程序局部性的近似；

WSSi (进程Pi 的工作集) = 最近访问的总的页面 △ (随时间变化)

  • 如果△太小将不能包含整个局部
  • 如果△太大将包含几个局部
  • 如果 △ = 无穷，将包含整个程序

总需求帧 D = ∑WSSi（局部近似）

如果 D > m，就会颠簸
策略：如果 D > m，则中止或换出其中的一个进程

缺页错误频率

缺页错误频率是一种比WSS更直接的方法

抖动具有高缺页错误率，因此可以设置上下限，使得：
如果缺页率太低，该进程降低物理帧数
如果缺页率太高，该进程增加物理帧数

与工作集策略一样，如果缺页错误率增加，并且没有空闲帧可用，我们可能不得不换出一个进程，将其交换到后备存储；然后，释放的帧被分发到缺页率较高的进程。

内存映射文件

内存映射文件或者I/O：通过将磁盘块映射到内存中的页面，允许将文件I/O视为常规内存访问

内存映射文件

实现机制

进入内存：

文件最初是使用请求分页读取的，请求分页之后，文件的一个页面部分从文件系统读入物理页面，在这之后，对文件的后续读/写操作被视为普通内存访问；

允许多个进程映射同一个文件，从而共享内存中的页面

返回磁盘：

周期性或在文件close()时，将写入的数据放入磁盘；

这样的话，访问文件是通过内存驱动文件I/O，而不是read()和write()系统调用，简化和加快了文件访问

Solaris举例

有些操作系统仅仅通过特定的系统调用来提供内存映射，而通过标准的系统调用来处理所有文件的IO；

也有的系统是不管文件是否指定为内存映射，都对文件进行内存映射；

Solaris为例，

如果一个文件指定为内存映射（使用系统调用mmap()指定），那么Solaris会将文件映射到进程地址空间；

如果一个文件通过普通的系统调用（open(), read(), write(), close()），系统仍然采用内存映射文件，只不过这里不是放到进程地址空间，而是内核地址空间，以加快访问速度；

共享内存

可以允许多个进程同时映射同一个文件，以便数据共享

可用于读/写非共享页面

共享内存可以通过内存映射文件实现（尽管也是通过单独的系统调用）

内存映射I/O

每个IO控制器包括保存命令和传输数据的寄存器，专用IO指令允许这些寄存器和系统内存之间进行数据传输；

为了方便，许多计算机提供了内存映射IO，具体实现方式是：使一组内存地址专门映射到设备寄存器，对这些内存地址的访问使得数据传送到到设备寄存器或者取自设备寄存器；

分配内核内存

分配内核内存指的是：当在用户模式下运行进程，这个进程请求额外的内存时，从内核维护的空闲页帧列表上分配页面；

用于分配内核内存的空闲内存池通常不同于用于普通用户模式进程的列表，原因是：

  • 内核需要为不同大小的数据结构分配内存，其中小于1页的会产生碎片；
  • 用户模式进程分配的页面不必位于连续的物理内存，而一些内核内存是需要连续的；

但是如果预先准备的页面未被使用，I/O和内存就会被浪费

假设预先准备了s个页面，并且被使用页面的比例为α

节省的 s*α 个缺页的成本 > 预先调 s*(1- α) 个不必要的页面的成本

α 接近于0 -> 预调页面失败；
α 接近于1 -> 预调页面成功；

页面大小

页面尺寸选择必须考虑：

碎片
页表大小
消除
I/O 开销
缺页数
位置
TLB的大小和有效性

总是 $2$ 的幂，通常在 $2^{12}$ (4,096字节)到 $2^{22}$ (4,194,304字节)的范围内

历史上是随着时间的推移而增长，即使对于移动系统也是如此

TLB范围

TLB范围：通过TLB可访问的内存量

TLB 范围 = (TLB Size) * (Page Size)

理想情况下，每个进程的工作集都存储在TLB，否则会有很高程度的缺页率

增加页面大小可能会导致碎片增加，因为并非所有应用程序都需要大的页面大小

提供多种页面尺寸，这使得需要更大页面大小的应用程序有机会使用它们，而不会增加碎片

程序结构

加载程序可以避免跨页边界放置例程，将每个例程完全保持在一页中。
多次相互调用的例程可以打包到同一个页面中

I/O 互锁

I/O互锁：有时必须将页面锁定到内存中

考虑I/O – 用于从设备复制文件的页面必须被锁定，不能被页面替换算法选择移除

固定（Pinning）页面以锁定到内存中
