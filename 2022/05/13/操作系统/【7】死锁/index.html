死锁问题 & 死锁特征

如果四个条件同时成立，就会出现死锁：

互斥

一次只有一个进程可以使用一个资源；

占有并等待

至少持有一个资源的进程正在等待获取其他资源持有的额外资源，即一个进程需要多个资源，他已经占有一个资源（或者更多），还想要另一个（或更多）；

非抢占

资源只能由他的进程再其完成其任务后主动释放；

循环等待

存在一组进程 {P0P_0P0​, P1P_1P1​,…, Pn−1P_{n-1}Pn−1​}，P0P_0P0​ 需要等待 P1P_1P1​ 所持有的资源，P1P_1P1​ 需要等待 P2P_2P2​ 所持有的资源，… ，Pm−2P_{m-2}Pm−2​ 需要等待 Pm−1P_{m-1}Pm−1​ 所持有的资源；

系统模型

系统模型是描述给进程分配资源过程的数学表达

定义

进程：P1,P2,...,PnP_1, P_2,...,P_{n}P1​,P2​,...,Pn​

资源类型：R1,R2,...,RmR_1, R_2,...,R_{m}R1​,R2​,...,Rm​

每个资源类型 RiR_iRi​ 有 WiW_iWi​ 个实例

每个进程使用如下资源：

  • 申请操作（P操作）
  • 使用操作（临界区）
  • 释放操作（V操作）

资源

物理资源（打印机，扫描仪）

逻辑资源（信号量、互斥锁、文件）

IPC

V 分为两部分：

  • 资源类型：R1,R2,...,RmR_1, R_2,...,R_{m}R1​,R2​,...,Rm​

小结

如果图没有环，则没有死锁；

如果图有环，1）如果每种资源类型只有一个示例，则死锁；2）如果每种资源类型有多个示例，则可能出现死锁

具体使用 死锁特征4 来判断

死锁处理方法

  1. 确保系统永远不会进入死锁状态：
    • 死锁预防
    • 死锁避免
  2. 允许系统进入死锁状态后恢复
  3. 忽略问题，假装系统中从未出现过死锁；这个方法被大多数操作系统使用，比如：Windows、Linux、UNIX；然后由应用程序开发人员编写处理死锁的程序

死锁预防

死锁预防也就是抑制死锁发生的 4 个必要条件

1. 抑制互斥

不要共享资源，必须持有非共享资源

某些资源实质上是不可共享的

2. 抑制保持并占有

必须保证当一个进程请求资源时，不持有其他资源；

也就是，直接拿所需的全部资源，不能全部都拿，那也不拿一部分，即，要不全拿，要不不拿；

缺点：资源利用率低，可能发生饥饿

3. 抑制非抢占

也就是采用抢占式的，即如果持有某种资源的进程请求另一个不能立即分配给他的资源，则释放这个进程持有的所有资源

被抢占的资源被添加到进程正在等待的资源中去

只有当进程能够重新获得他的旧资源以及他正在请求新的资源时，他才会重新启动

该协议通常应用于状态可以很容易保存和恢复的资源，例如 CPU 寄存器和内存空间

4. 抑制循环等待

强制所有资源类型的总顺序，并要求每个进程以递增的顺序请求资源（让这个环连不回来）

换句话说，为了请求资源 RjR_jRj​，进程必须首先释放所有的 RiR_iRi​，其中 i≥ji \ge ji≥j

死锁预防使得 设备利用率 和 吞吐量 降低

死锁避免

条件比死锁预防更宽松一些；

死锁避免即：把资源给一个线程的之前，判断这个分配合不合理，会不会出现死锁，会出现的话就不分配了

死锁避免需要系统有一些先验知识

  • 要求每个进程声明他可能需要的每种类型的资源的最大数量；
  • 资源分配状态由可用资源数量、已分配资源数量以及进程最大需求资源数量来定义；
  • 死锁避免算法采用动态检测资源分配状态，以确保永远不会出现循环等待状态；

当一个进程请求所用资源时，系统判断立即分配是否能使系统处于安全状态

安全状态是指：针对所有进程，存在一个时间序列，代表了哪个进程什么时候开始、什么时候结束，使得完成这些进程不会死锁，这个序列称为安全序列；

详细解释安全状态： 如果系统存在所有进程的序列 <P1,P2,...,Pn><P_1, P_2, ..., P_n><P1​,P2​,...,Pn​>，使得对于每个 PiP_iPi​，PiP_iPi​ 仍然可以请求的资源可以由当前可用资源 + 所有Pj(1≤j<i)P_j(1 \le j <i)Pj​(1≤j<i)持有的资源来满足，则系统处于安全状态

即：

  • 如果PiP_iPi​资源需求不是立即可用的，那么PiP_iPi​可以等到直到所有的PjP_jPj​都完成；
  • 当PjP_jPj​完成时，PiP_iPi​可以后的所需的资源，执行，返回分配的资源，并终止；
  • 当PiP_iPi​终止时，Pi+1P_{i+1}Pi+1​可以获得自己需要的资源，以此类推；

银行家算法 —— 死锁避免的一个算法

前提

多个资源多个实例；
每一个资源必须先验的地推算出最大资源的需求量；
当一个进程申请资源时，他可能需要等待；
当一个进程得到其所有申请资源后，他必须在有限的时间内归还；

死锁检测和死锁恢复

死锁检测

条件相比死锁预防和死锁避免更加宽松，允许进入死锁状态，进入之后检测到了再恢复；

维护一个等待图， 等待图就是资源分配图的简化版本（去掉了资源结点，只保留了进程结点）；
定期调用算法在图中搜索循环的算法。如果有循环，则存在死锁；
检测图中循环的算法需要 n2n^2n2 次运算，其中 nnn 是图中顶点的数量，也就是进程的个数；

检测算法流程

定义数据结构：
Available：长度为 m 的向量表示每种类型的可用资源数量
Allocation：一个 n * m 矩阵定义了当前分配给每个进程的每种类型的资源数量
Request：n * m 矩阵表示每个进程的当前请求。如果Request [i][j] = k, 那么进程 Pi 正在请求资源 Rj 的 k 个实例

具体算法流程：

  1. 令Work 和Finish 分别为长度为m 和n 的向量 , 初始化：
    (a) Work = Available
    (b) For i = 1,2, …, n, 如果Allocationi ≠ 0, 则 Finish[i] = false; 否则, Finish[i] = true

  2. 找到一个索引 i 使得:
    (a) Finish[i] == false
    (b) Requesti \le Work
    如果没有这样的 i 存在，则转到步骤 4

  3. Work = Work + Allocationi;
    Finish[i] = true 则转到步骤2

  4. 如果 Finish[i] == false，对于某些i，1 \le i \le n，则系统处于死锁状态. 并且, 如果Finish[i ] == false， 则 Pi 死锁

算法需要 O(m∗n2)O(m * n^2)O(m∗n2) 次操作来检测系统是否处于死锁状态

检测算法的使用

调用的时间和频率取决于：

  • 多久可能发生一次死锁？
  • 当死锁发生时，有多少进程会受到影响？

为每个资源请求调用死锁检测算法

  • 可观的开销；
  • 告诉死锁进程中的哪一个“导致”了死锁；

以定义的时间间隔调用算法

  • 如果任意调用检测算法，资源图中可能会有许多循环，因此我们无法判断死锁进程中的哪一个“导致”了死锁。

死锁恢复 —— 进程终止

终止所有的死锁进程；

一次终止一个进程，知道死锁循环被消除（必须每次调用死锁检测算法来确定是否有任何进程扔处于死锁状态）；

应该按照哪个顺序终止？

  1. 进程的优先级；
  2. 计算进程需要多长时间，还有多长时间才能完成；
  3. 进程使用的资源；
  4. 进程完成还需要多少资源；
  5. 需要终止多少个进程；
  6. 进程是交互的还是批处理的；

死锁恢复 —— 资源优先级

选择一个受害者 —— 最小代价化（死锁进程持有的资源数量以及该进程迄今为止消耗的时间）

回滚 —— 返回某个安全状态，重新启动该状态的进程

  • 完全回滚：中止进程，然后重新启动
  • 尽管仅在必要时回滚进程以打破死锁更有效，但这种方法要求系统保留所有相关的正在运行的进程状态的更多信息

饥饿 – 相同的进程可能总是被选为受害者，包括成本因素中的回滚次数
