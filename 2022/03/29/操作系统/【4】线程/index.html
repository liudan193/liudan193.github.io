【4】线程

发表于2022-03-29|更新于2022-04-01|操作系统
|阅读量:

一、概述

传统的进程只能控制以1个线程 —— 1个程序计数器，1个可执行的指令序列

1个线程是CPU执行1个基本单元，包括了：

  • 1个程序计数器
  • 1个栈
  • 1组可使用的寄存器（以及1个线程ID）

1.1 多线程程序是指一个单独进程中包含了多个线程：

  • 每个线程拥有单独的：程序计数器，栈和寄存器组
  • 不同线程之间共享：通用代码、数据以及特定的数据结构（比如打开的文件列表等）

1.2 使用多线程的动机：

  • 进程中存在多个任务单独执行的情况，一个应用程序通常是作为一个具有多个控制线程的独立进程实现的
  • 应用程序的多个任务可以由单独的线程实现
    • 某一任务被阻塞，其他任务的执行不受影响
    • 例如：网络浏览器程序可能有一个线程用于显示图像和文本，另一个线程用于从网络中接收数据；
  • 进程创建是重量级的，而线程的创建是轻量级的
  • 可以简化代码，提高效率
  • 内核通常是多线程的

1.3 优点：

  • 及时响应 ———— 部分线程被阻塞，其他可继续执行（对用户界面来说尤其重要）
  • 资源共享 ———— 线程共享进程资源，比共享内存或消息传递更容易
  • 经济 ———— 比线程创建便宜，线程切换的开销比上下文切换低
  • 可扩展性 ———— 线程可以在多处理器上并行运行，而单线程只能运行在一个 CPU 上

二、多核编程

区分并发和并行：

多线程程序在单核CPU中执行时不同线程之间有时间间隔（并发执行 Concurrent）

多线程程序在多核CPU中可以同时在多个可用的核中执行真实的并行处理（Parallel Processing）

2.1 多核编程及其挑战

并行性意味着一个系统可以同时执行多个任务

  • 在多处理器 / 多核系统中，有特定的调度算法来充分利用多处理器 / 多核

并发性系统支持多个任务，允许所有任务都能取得进展

  • 在单处理器中，提供并发性的调度程序

多线程越来越受欢迎，CPU采用超线程技术来允许单核中支持更多的并行线程（hyperthreading）

多线程编程带来的挑战：

  • 识别任务 —— 确认任务中哪些子任务可以并行 / 并发执行
  • 平衡 —— 确认并行子任务价值相同，无需在无价值任务上浪费线程资源
  • 数据分割 —— 防止线程间相互影响
  • 数据依赖 —— 线程间数据相互依赖，需要进行同步处理
  • 测试和调试 —— 在并发和并行情况下更加复杂和困难，难以定位竞争条件

2.2 并行的类型

  • 数据并行 —— 将相同数据的子集分布在多个计算核上，每个核执行相同的操作
  • 任务并行 —— 跨内核分配线程，每个线程执行独特的操作
  • 混合并行

2.3 Amdahl 定律

  • 该公式确定由于额外计算核的增加潜在地改进性能
  • 如果 S 是公式的一部分，他在具有 N 个处理核的系统上可以串行执行，那么该公式如下：$$ 加速比 \le \frac{1}{S + \frac{1-S}{N} }$$
  • 也就是说，如果应用程序的 75% 并行而 25% 串行，在具有两格处理核的系统上运行这个程序，能得到 1.6 倍的加速比
  • 随着 N 趋于无穷大，加速比收敛到 $\frac{1}{S}$
  • 对于通过增加额外计算核而获得的性能，应用程序的串行部分可能具有不成比例的效果

三、多线程模型

现代操作系统中的线程

  • 用户线程
    • 内核层上，应用程序中使用的并行 / 并发程序，由用户级线程库管理
  • 内核线程
    • 操作系统内核中，支持多个任务并行执行内核系统调用
    • 大多数通用操作系统都支持
    • 在某些实现中，用户线程必须映射到内核线程执行

3.1 多对一模型

多对一模型：映射多个用户级线程到一个内核线程

  • 一个线程阻塞导致整个进程阻塞
  • 多线程不能在多核系统上并行运行，因为任一时间只有一个线程可以访问内核
  • 目前很少有系统使用这个模型

3.2 一对一模型

一对一模型：映射每个用户进程到一个内核进程

优点：

  • 该模型在一个线程执行阻塞系统调用时，能允许另一个线程继续执行，提供了更好的并发功能

缺点：

  • 创建一个用户线程就要创建一个响应的内核线程
  • 提供比多对一更高的并发性
  • 创建内核线程的开销会影响应用程序的性能，所以限制了系统的线程数

例子：

  • Windows
  • Linux
  • Solaris 9 及更高版本

3.3 多对多模型

多对多模型：将多个用户级线程映射到多个（和用户线程同样数量或者更小数量）内核线程

  • 允许操作系统创建足够多的内核线程（一对一模型的缺点）
  • 能在多处理器系统上并发执行（多对一模型的缺点）
  • 例如：Solaris 9 之前的版本支持这种模型

3.4 双层模型

双层模型：多对多模型变种，既允许多个用户线程映射到多个（和用户线程同样数量或者更小数量）内核线程，又允许绑定某个用户线程到一个内核线程

四、线程库

线程库为程序员提供创建和管理线程的 API

两种主要的实现方法：

  • 用户级线程库：在用户空间中提供一个没有内核支持的库
    • 此库的所有代码存放在用户空间中，调用库函数知识导致用户空间的一个本地函数调用，而不是系统调用
  • 内核级线程库：实现由操作系统直接支持的内核级的一个库
    • 代码和数据结构存在于内核空间中，调用库函数通常会导致对内核的系统调用

常见线程库：

  • POSIX Pthread 库
  • Win 32 线程库
  • Java Thread 库

五、隐式多线程

随着线程数量的增加，隐式线程越来越受欢迎，显式线程的程序正确性难以达到

由编译器和运行库时库来创建和管理线程，而不是程序员

探讨了三种方法：

  • 线程池
  • OpenMP
  • 大中央调度

5.1 线程池

线程池：在进程开始时创建一定数量的线程，并放到池中以等待工作

优点：

  • 使用现有的线程比等待创建一个新地线程更快；
  • 线程池限制了在任何时候可用线程的数量。这对那些不能支持大量并发线程的系统非常重要；
  • 将要执行任务从创建任务的机制中分离出来，允许采用不同策略运行任务；

当线程池满时，进程需要等待有线程空闲出来；

线程池可容纳线程的最大数量可以动态调整，以满足不同任务的需求；

六、多线程问题

6.1 系统调用 fork() 和 exec()

fork() -> 程序内的某个线程调用 fork()，那么复制出来的新进程是复制旧线程的所有线程，还是新进程新创建独立的线程（只有单个线程）？

一些 UNIX 系统有两个版本的 fork()，一种复制了所有的线程，另一种只复制了调用 fork() 的线程；

exec() -> 替换正在进行的进程，包括所有的线程；

6.2 信号处理

信号：信号在 UNIX 系统中用于通知进程已经发生了某个特定的事件

信号处理程序用于处理信号：

  • 信号由特定事件产生
  • 产生的信号要发送给进程
  • 信号一旦发送，必须加以处理，信号由两个信号处理程序之一处理：默认 或者 用户自定义

每个信号都有一个默认信号处理程序，由内核来运行

  • 用户自定义信号处理程序可以改写默认信号处理程序

对于单线程程序的信号处理，信号总是发送给进程；而对于多线程，情况较为复杂；

6.3 线程撤销

在线程完成之前终止线程

要撤销的线程成为目标线程

目标线程的撤销通常有两种情况：

  • 异步撤销：立即终止目标进程
  • 延迟撤销：允许目标线程定期检查是否应该取消该线程

调用线程撤销请求撤销目标进程，但实际上线程的取消取决于线程状态：
|模式|状态|类型|
|-|-|-|
|关闭|禁用|-|
|延迟|启用|延迟|
|异步|启用|异步|

如果线程撤销已被弃用，则撤销请求将保持挂起状态，直到线程稍后弃用撤销并且响应这个请求

在 Linux 系统上，线程撤销是通过信号来处理的

6.4 线程本地存储

线程本地存储

TLS是一种在多线程时使用的技术，它可以使你的全局变量、静态变量以及局部静态、静态成员变量成为线程独立的变量，即每个线程的TLS变量之间互不影响。

线程本地存储（TLS）允许每个线程拥有自己的数据副本

当无法控制线程创建过程（即使用线程池）时很有用

与局部变量不同

  • 局部变量只在单个函数调用时才可见
  • TLS 数据在多个函数调用时都可见

类似于静态数据：TLS对于每个线程都是唯一的

6.5 调度程序

多对多模型和双层模型都需要内核与线程库互相通信来维护分配给应用程序适当数量的内核线程，以达到更好的性能；

解决方案：在用户和内核线程之间增加一个中间数据结构 —— 轻量级进程（LWP）

  • LWP 表现为虚拟处理器，以便应用程序调度并运行用户线程
  • 每个 LWP 与一个内核线程相连

调度器激活提供回调

  • 一种从内核将有关特定事件通知应用程序，有线程库通过回调处理程序来处理的通信机制

clone()允许子任务共享父任务的地址空间
