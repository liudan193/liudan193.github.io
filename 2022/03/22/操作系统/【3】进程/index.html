进程的概念

1 进程：一个正在执行的程序，进程的执行必须按照顺序进行

2 操作系统执行多种程序：

批处理系统 —— 作业
分时系统 —— 用户程序和任务

3 进程的多个部分：

内存：存放程序代码、栈、数据段、堆；
当前活动：当前状态、程序结束器、处理器寄存器，以及上下文信息（即所有寄存器当前的数据值）

4 程序 VS 进程：

程序是存储在磁盘上的被动实体，进程是主动的，当一个可执行文件被加载到内存时，这个程序就成为进程；
一个程序可以是多个进程（比如：多个用户执行同一程序，一个程序执行多个任务）

5 进程控制块（PCB）

进程控制块（PCB）：包含许多与每个进程相关的信息

主要包括：
进程状态 —— 运行、等待、停止等；
进程ID —— 进程编号
程序计数器 —— 下个要执行指令的位置
CPU寄存器 —— 所有与进程相关寄存器的内容
CPU调度信息 —— 进程优先级、调度队列指针和其他调度参数
内存管理信息 —— 内存页表、段表等
记账信息 —— CPU时间、实际使用时间、时间期限、记账数据、作业或进程数量等
I/O状态信息 —— 分配给进程的IO设备列表、打开的文件列表等

C语言操作 Linux

创建进程有两种方式，一是由操作系统创建；二是由父进程创建。操作系统创建的进程，它们之间是平等的，一般不存在资源继承关系。而由父进程创建的进程（子进程），它们和父进程存在隶属关系。子进程又可以创建进程，形成一个进程家族。

系统调用fork是创建一个新进程的唯一方法。进程调用fork函数就创建了一个子进程。

创建了一个子进程之后，父进程和子进程争夺CPU，抢到CPU者执行，另外一个挂起等待。若要父进程等待子进程执行完毕后再继续执行，可以在fork操作之后调用wait或waitpid。

fork函数

fork() 函数调用后有2个返回值，调用一次，返回两次。成功调用 fork 函数后，当前进程实际上已经分裂为两个进程，一个是原来的父进程，另一个是刚刚创建的子进程。

fork() 函数的2个返回值，一个是父进程调用 fork 函数后的返回值，该返回值是刚刚创建的子进程的 ID；另一个是子进程中 fork 函数的返回值，该返回值是 0。这样可以用返回值来区分父、子进程。

fork(): 父子进程不共享一段地址空间，修改子进程，父进程的内容并不会受影响。

进程的调度

1. 调度的基本概念

调度的目的：

  1. 最大限度地利用 CPU
  2. 在可接受的响应时间内在进程之间快速切换 CPU，尤其是需要和用户交互的进程

进程调度器可以从多个可用进程队列中选择一个进程到 CPU 上执行

维护进程调度的队列：

  1. 作业队列 —— 系统中所有进程的集合
  2. 就绪队列 —— 保存驻留在内存中的、就绪的、等待运行的进程
  3. 设备队列 —— 一组等待IO设备的进程列表
  4. 进程在各个队列之间迁移

2. 调度程序

1. 短期调度器 —— 选择接下来要执行的进程，并分配 CPU

  1. 有时候是系统中唯一的调度程序
  2. 短期调度程序被频繁调用 -> 必须快速

2. 长期调度器 —— 选择那些程序应该进入就绪队列

  1. 长期调度程序的调度并不频繁
  2. 长期调度器控制多道程序粒度（内存中的进程数量）
  3. 分时系统，比如 UNIX 和 Windows 通常不包括长期调度器，多用在批处理中

3. 中期调度程序

根据任务类型的不同，进程可以分为：

  • IO 密集型进程 —— 执行IO时间比执行计算时间花费更多，有许多短暂的 CPU 突发；
  • CPU 密集型进程 —— 花费更多的时间在计算上，少有很长的 CPU 突发

中期调度程序：从内存中删除进程，移动到磁盘上，并从中断处继续执行（执行交换动作）

4. 上下文切换

当 CPU 切换到另一个进程的时候，系统必须保存旧进程的状态，并通过上下文切换为新进程加载其保存的状态

PCB保存程序的当前情况以及控制进程运行的全部信息（即所有寄存器和程序计数器信息）

上下文切换的时间是纯开销，因为在切换时系统没有做任何有用的工作

操作系统和 PCB 越复杂，上下文切换时间越长
上下文切换时间越短越好

进程的操作

进程的创建和删除

1. 进程的创建

1. 父子进程

父进程创建子进程，子进程又创建其他进程，形成进程树

通过 进程标识符（pid） 识别和管理进程

资源共享选项（CPU、内存、文件 IO 设备）：

  • 父进程和子进程共享所有的资源
  • 子进程获得父进程资源的子集
  • 父进程和子进程不共享资源
  • *执行选项**：
  • 父进程等待，直到某个或全部子进程执行完
  • 父进程和子进程并发执行

父子进程执行的详细解释：

  1. 父进程等待，直到某个或者全部子进程执行完
    • 父进程使用wait()系统调用，等待子进程结束后继续执行
    • UNIX shell 通常会等待子程序结束后再执行新任务
  2. 父进程和子进程并发执行
    • 父进程继续执行程序
    • UNIX shell 以后台进程形式执行过程
    • 有时候父进程会执行一段时间，处理完自己的任务后，再调用wait()等待子进程处理，如：创建多个子进程，然后再等待的情况

2. 进程创建

地址空间：

  • 子进程是父进程的复制品（它具有与父进程同样的程序和数据）
  • 子进程加载另一个新程序

2. 进程的终止

原因①. 进程执行完最后一条语句，通过系统调用exit()请求操作系统删除自身

  1. 返回状态值到父进程，整数 0 表示正常结束，其他整数表示异常结束

  2. 进程的资源由操作系统释放

原因②. 父进程可以使用系统调用kill()终止子进程的执行，这样做的原因：

  • 子进程适应了超过它所分配的资源
  • 不再需要分配子进程的任务
  • 父进程正在退出，而且操作系统不允许无父进程的子进程继续执行

原因③. 除了以上情况，有时候为了回复某些未处理的中断也会终止响应进程

某些操作系统不允许子进程在父进程已经终止的情况下存在。如果一个进程终止，那么他的所有子进程也必须终止：

  • 级联式终止：所有的子进程、子进程的子进程等都被终止
  • 终止由操作系统发起

父进程可以通过系统调用 wait() 等待子进程的终止，并返回状态信息和终止进程的 pid

当子进程已经终止，但是其父进程没有调用 wait() ，这样的子进程称为僵尸进程

  • 当进程终止时，他的资源将被系统回收，但是进程表（process table）中仍然保留了子进程的退出状态信息，直到父进程使用系统调用 wait() 返回后清除

如果父进程没有调用 wait() 就终止，子进程就会变成孤儿进程

  • 孤儿进程由systemXX进程接受，接着运行

进程间通信

1. 进程协作概念

操作系统内并发的进程可以是独立的，也可以是协作的

协作的进程能影响其他进程或受其他进程所影响，包括共享数据

进程协作的目的：

  • 信息共享：并行进程共享数据
  • 计算加速：将复杂的任务划分为子任务并行执行
  • 模块化：将复杂系统拆分为独立协作的不同部分，提高效率，减小故障影响，便于维护
  • 方便

2. 通信模型

协作进程需要一种进程间通信（IPC）

两种基本模型：

  • 共享内存
    • 速度快，适合在同一计算机上大量数据的传输
    • 配置比较复杂，不适合跨计算机
  • 消息传递
    • 速度比共享内存慢
    • 适合少量数据传输，适合跨计算机传输

共享内存：

  • 通常来说，操作系统尽力阻止跨进程的内存访问
  • 共享内存机制允许一个进程在内存中创建一块供其他进程访问的区域，实现进程间消息的传递；当其他进程访问这些消息时，首先需要将共享内存区域的地址映射到自己的地址空间中
  • 通过在共享区域中的读写操作实现信息的传递
  • 主要难点是不同进程间读写时的同步问题

3. 生产者-消费者问题

进程通信的范例：生产-消费者模型

  • 生产者进程生成(produce)信息， 以供消费者进程消费(consume)信息

模型类型

  • 无界缓冲区对缓冲区的大小没有实际限制
  • 有界缓冲区假设有一个固定大小的缓冲区；生产者只有当缓冲区有空闲区域时才能生成信息；消费者只有当缓冲区中数据准备好时才能消费信息

4. 共享内存

  • 需要通信的进程之间建立共享内存区域
  • 通信受控于用户进程，而非操作系统
  • 主要问题是提供一种机制，允许用户进程在访问共享内存时同步它们的操作

5. 消息传递

如果进程P和Q希望进行通信，它们需要：

  • 在它们之间建立 通信链路
  • 通过 发送/接收（send/receive） 交换信息

5.1 如何建立链路

  • 物理上：
    • 共享内存
    • 硬件总线
    • 网络
  • 逻辑：
    • 直接或间接的通信
    • 同步或异步的通信
    • 自动或显式的缓冲

5.2 直接通信和间接通信的解释

直接通信
  • 需要通信的每个进程必须明确指定通信的接受者或发送者
    • send(P, message) - 向进程P发送消息
    • receive(Q, message) - 从进程Q接收消息
  • 通信链路具有以下属性：
    • 需要通信的每对进程之间，自动建立链路
    • 每个链路只与两个进程相关
    • 每对链路之间只有一个链路
    • 链路可能是单向的，但通常是双向的
间接通信
  • 消息从邮箱（也称为端口）发送和接收
    • 每个邮箱都有一个唯一的 id
    • 进程只有在共享邮箱的时候才能通信
  • 通信链路的特点：
    • 只有在两个进程共享一个邮箱时，才能建立通信链路
    • 一个链路可以与多个进程相关联
    • 每对进程可以共享多个通信链路，每个链路对应一个邮箱
    • 链路可以是单向或双向的
  • 操作
    • 创建新的邮箱
    • 通过邮箱发送和接收消息
    • 删除邮箱
  • 原语定义如下：
    • send(A, message) - 向邮箱 A 发送消息
    • receive(A, message) - 从邮箱 A 接收消息
  • 解决一些问题
    • 比如说，进程 P1、P2 和 P3 共享邮箱 A，P1 发送消息 P2、P3 接收，那么谁将收到消息？
    • 解决：
      • 允许一个链路最多和两个进程关联
      • 允许一次最多一个进程执行 receive() 操作
      • 允许系统随意选择一个进程接收消息，并通知发送方接收者是谁

5.3 同步（Synchronization）

消息传递可以是阻塞（blocking）的，也可以是非阻塞（non-blocking）的

阻塞被认为是同步的：

  • 阻塞发送 – 发送进程阻塞，直到消息由接收进程或邮箱所接收
  • 阻塞发送 – 接收进程阻塞，直到有消息可用

非阻塞被认为是异步的：

  • 非阻塞发送 – 发送进程发送消息，并且恢复操作
  • 非阻塞接收 – 接收进程收到一个有效消息或空消息

5.4 缓存（buffering）

通信进程交换的消息总是驻留在缓冲队列中

缓冲队列三种类型：

  1. 零容量——没有消息在链路上排队。发送者应阻塞， 直到接收者接收到消息
  2. 有限容量——有限长度的 n 条消息，如果链路已满，发送方必须等待
  3. 无限容量——无限长度，发送方从不等待

IPC 系统示例 – POSIX

POSIX 共享内存

  1. 进程首先创建共享内存对象（也用于打开现有对象以便于共享）：
    shm_fd = shm_open(name, O CREAT | O RDWR, 0666);

  2. 设置对象的大小：
    ftruncate(shm fd, 4096);

  3. 现在对象可以写入共享内存：

客户机—服务器通信（主要是管道）

管道允许两个进程进行通信

问题：
  管道允许单向通信还是双向通信？
  如果允许双向通信， 它是半双工还是全双工？
  通信进程之间是否应有一定的关系（如父子关系）？
  管道通信能否通过网络， 还是只能在同一台机器上进行？

普通管道—只能由创建进程访问。通常，父进程创建一个管道，并使用它与父进程创建的其它子进程通信
命名管道—可以在没有父子关系的情况下访问

1. 普通管道

  • 普通管道允许两个进程以标准的生产者—消费者方式进行通信
  • 生产者写到一端（写入端）；消费者从另一端（读取端）读取
  • 因此普通管道是单向的；实现双向通信需要创建2个管道
  • 需要通信进程之间的父子关系

2. 命名管道

  • 命名管道比普通管道更强大
  • 通信是双向的
  • 通信进程之间不需要父子关系
  • 多个进程可以使用命名管道进行通信
  • UNIX 和 Windows 系统支持命名管道

UNIX系统中，命名管道(FIFOs)在文件系统中以文件形式存储在系统中

使用`mkfifo()`调用进行创建，使用`read()`, `write()`, `open()` 调用进行操作
UNIX的命名管道时双向的，但也是半双工的，仍需要建立2个管道来进行双向通信
命名管道只能在同一台机器上使用，跨机器通信需要使用Socket套接字
